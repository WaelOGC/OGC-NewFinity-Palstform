# OGC NewFinity — Challenge Content & Publishing Guidelines (v1.0)

## 1. Introduction

This document defines the official standards, structure, formatting rules, quality requirements, and publishing workflows for all Challenge Content across the OGC NewFinity platform.

These guidelines ensure:

- Consistent challenge quality
- Clear structure for participants
- Fair competition rules
- Safe and compliant content
- Scalable contribution management
- Standardized global formatting

This applies to all challenge categories, difficulty levels, formats, and age groups.

## 2. Challenge Content Structure

Every challenge must follow the standardized structure:

### 2.1 Challenge Metadata

- Challenge ID
- Title
- Category
- Difficulty level
- Age group (if applicable)
- Duration
- Submission type
- Reward type

### 2.2 Public Description

Clear and motivating summary of:

- Purpose
- Required task
- Expected outcomes

### 2.3 Rules & Requirements

Must include:

- Allowed actions
- Forbidden actions
- Submission restrictions
- Scoring rules
- Disqualification conditions

### 2.4 Submission Instructions

Required details:

- Acceptable file types
- File size limits
- Naming conventions
- Submission deadline
- Optional preview attachments

### 2.5 Evaluation Criteria

Judging must be based on:

- Accuracy
- Creativity
- Completeness
- Technical quality
- Alignment with theme

### 2.6 Reward Structure

Rewards may include:

- OGCFinity
- Badges
- Certificates
- Ranking points

## 3. Content Categories & Templates

### 3.1 Standard Challenge Categories

- Creative Challenges
- STEM Challenges
- AI & Coding
- Writing & Research
- Problem-Solving
- Art & Media
- Community Engagement

### 3.2 Category Template Example

```yaml
title: "AI Art Concept Challenge"
category: "AI & Creativity"
difficulty: "Intermediate"
description: "Create an original AI-generated artwork using your favorite model..."
```

### 3.3 Per-Category Requirements

Each category may include additional rules for:

- Tools
- Techniques
- Allowed resources
- Format constraints

## 4. Publishing Workflow

### 4.1 Roles

- Content Author — drafts challenge
- Editor — reviews clarity & grammar
- Technical Reviewer — validates rules & scoring
- Compliance Officer — checks policy alignment
- Publisher — final approval

### 4.2 Publishing Steps

- Draft submitted
- Internal review
- Technical verification
- Compliance & safety check
- Challenge preview generated
- Publish to platform
- Notify relevant audience groups

### 4.3 Localization Workflow

Challenges must be translated before publishing into:

- EN
- AR
- FR
- ES

Translation follows Document 112 standards.

## 5. Writing Standards & Tone

### 5.1 Tone Requirements

Challenge descriptions must be:

- Clear
- Motivating
- Educational
- Inclusive
- Positive

### 5.2 Grammar & Style

- Use active voice
- Avoid overly technical language
- Short sentences preferred
- Avoid ambiguity
- Use consistent terminology

### 5.3 Formatting Rules

- Use Markdown
- H2 for sections, H3 for subsections
- Bullet lists for rules
- Tables for scoring breakdown

## 6. Safety & Compliance

### 6.1 Prohibited Content

Challenges must not include:

- Harmful or dangerous activities
- Offensive themes
- Political or religious bias
- Illegal instructions
- Personal data requests
- Age-inappropriate content

### 6.2 Youth Safety

If targeting children:

- Age-appropriate wording
- Clear parental guidance notes
- No external unsafe links

### 6.3 Fair Competition Requirements

- Rules must be explicit
- Scoring must be transparent
- No hidden criteria
- No discriminatory evaluation factors

## 7. Accessibility Requirements

Challenges must be accessible to:

- Screen readers
- RTL languages
- Low-vision users

Rules:

- Alternative text for images
- Clear headings
- High-contrast formatting
- Avoid color-only indicators

## 8. SEO & Discoverability

### 8.1 SEO Formatting

Each challenge must include:

- SEO title
- Meta description
- Relevant keywords
- Open Graph image

### 8.2 Content Linking

Challenges should link to:

- Related challenges
- Knowledge base articles
- Tutorials
- Previous challenge winners (optional)

## 9. Content Storage & Versioning

### 9.1 Storage Format

Challenges stored in:

- YAML for structured fields
- Markdown for descriptions

### 9.2 Version Control

All edits tracked via:

- Version number
- Updated timestamp
- Editor name
- Change log

### 9.3 Draft vs Published States

- Draft → editable
- Published → locked
- Deprecated → archived

This document defines the Challenge Content & Publishing Guidelines for OGC NewFinity, ensuring a consistent, safe, and engaging experience for participants around the world.
